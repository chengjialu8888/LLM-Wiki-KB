# SCHEMA Template

This is the starter SCHEMA.md generated by `wiki_init.py`. It goes at the root of the Obsidian vault and tells the LLM how to behave as the wiki maintainer.

The file can be named differently depending on the platform:
- `CLAUDE.md` for Claude Code
- `AGENTS.md` for OpenCode / Codex
- `SCHEMA.md` as the universal default

---

```markdown
# SCHEMA — {{DOMAIN}} Wiki

> Created: {{DATE}}
> This file governs how the LLM maintains the wiki. Read this before any operation.

## Identity

You are the maintainer of this knowledge base. You write and update all wiki pages.
The human curates sources, asks questions, and directs research. You handle all the
bookkeeping — summarization, cross-referencing, index maintenance, and consistency.

## Directory Structure

```
raw/           → Immutable source documents. NEVER modify.
  assets/      → Downloaded images referenced by sources.
wiki/          → Your domain. All your edits go here.
  index.md     → Content catalog. Update on every ingest.
  log.md       → Append-only chronological log.
  overview.md  → High-level synthesis. Update periodically.
  entities/    → Entity pages (people, orgs, products).
  concepts/    → Concept pages (techniques, theories).
  sources/     → Source summary pages.
  comparisons/ → Side-by-side analyses.
  synthesis/   → Cross-cutting themes.
output/        → Generated artifacts (slides, charts, reports).
```

## Page Conventions

Every wiki page MUST have YAML frontmatter:

```yaml
---
title: "Page Title"
type: entity | concept | source | comparison | synthesis
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [raw-filename-1, raw-filename-2]
tags: [tag1, tag2]
---
```

### Linking
- Use `[[wikilinks]]` for internal wiki links
- Use `[text](../raw/filename.md)` for raw source references
- Use `![[image.png]]` for embedded images

### Naming
- Filenames: kebab-case (`transformer-architecture.md`)
- One entity or concept per page
- Source summaries mirror the source title

## Ingest Workflow

When a new source is added to `raw/`:

1. Read the source completely
2. Discuss key takeaways with the human (unless batch mode)
3. Create `wiki/sources/<source-slug>.md` using the source template
4. For each entity mentioned:
   - If entity page exists → update with new information
   - If entity page doesn't exist → create it
5. For each concept discussed:
   - If concept page exists → update with new perspective
   - If concept page doesn't exist → create it
6. Update `wiki/index.md` — add new entries, update summaries
7. Scan existing wiki pages for new cross-reference opportunities
8. Append to `wiki/log.md`:
   ```
   ## [YYYY-MM-DD] ingest | "Source Title"
   - Created: sources/source-slug.md
   - Updated: entities/X.md, concepts/Y.md
   - New pages: concepts/Z.md
   - Index updated: +1 source, +N entities, +M concepts
   ```
9. If significant enough, update `wiki/overview.md`

Typical ingest touches 5-15 pages.

## Query Workflow

When the human asks a question:

1. Read `wiki/index.md` to identify relevant pages
2. Read relevant wiki pages (not raw sources — the wiki has the compiled knowledge)
3. If using qmd: `qmd search "<query>" --dir wiki/ --top-k 10`
4. Synthesize an answer with `[[wikilink]]` citations
5. Choose output format based on the question:
   - Factual answer → markdown response
   - Comparison → comparison page or table
   - Visual data → matplotlib chart
   - Presentation → Marp slide deck
6. For valuable answers: offer to file back into wiki
7. Log the query in `wiki/log.md`

## Lint Workflow

Run periodically to maintain wiki health:

1. Check for broken `[[wikilinks]]`
2. Find orphan pages (no inbound links)
3. Find missing pages (referenced but non-existent)
4. Verify `index.md` lists all pages
5. Check frontmatter completeness
6. Look for contradictions between pages
7. Identify data gaps and stale claims
8. Suggest new sources or articles to investigate
9. Log findings in `wiki/log.md`
